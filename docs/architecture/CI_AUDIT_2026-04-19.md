# CI Audit 2026-04-19 — CA2007 and vitest mock fallout

**Status:** Fixed on 2026-04-20 · **Owner:** Steve Ackley

Short writeup of two CI failures that landed together on main and blocked every Dependabot PR.

## 1. Backend — CA2007 treated as error in `StackAlchemist.Engine`

The root `.editorconfig` sets `dotnet_diagnostic.CA2007.severity = error` for all `.cs` files. That rule ("Consider calling ConfigureAwait on the awaited task") exists to guard library code from deadlocking when awaited on a captured `SynchronizationContext`.

**Why it's noise here:** `StackAlchemist.Engine` is an ASP.NET Core Web API. ASP.NET Core has no `SynchronizationContext` — `ConfigureAwait(false)` is a no-op. Sprinkling it across every `await` in the Engine project adds clutter without changing runtime behavior.

**Fix:** `src/StackAlchemist.Engine/.editorconfig` overrides the root rule to `severity = none` for that project only. Libraries (none shipped yet) and the Worker keep the strict rule by default.

**Not addressed:** CA1848 (LoggerMessage delegates) still warns — kept as warning because it's a real perf suggestion we may want to act on later.

## 2. Frontend — vitest IntersectionObserver mock

`vitest.setup.ts` mocked `IntersectionObserver` via `vi.fn().mockImplementation(() => ({...}))`. Arrow functions are not constructors, so `new IntersectionObserver(...)` (called from Next.js `use-intersection` under `next/link` prefetch) throws `TypeError: ... is not a constructor` in CI.

Local runs passed because the code path wasn't exercised — `next/link` prefetch behavior depends on IO timing the CI runner hits but a local dev run does not.

**Fix:** replaced the arrow-function mocks with real `class` declarations assigned to `global.IntersectionObserver` and `global.ResizeObserver`. Classes are constructable; shape still satisfies the interfaces Next/React Flow probe.

## Operational lesson

`npx vitest run --reporter=junit` replaces the default reporter, so failures only land in `test-results/junit.xml` — stdout shows nothing. CI's `vitest.config.ts` already specifies `reporters: ['default', 'junit']`, so fixing the workflow to drop `--reporter=junit` from the CLI flag (let the config drive it) would surface future test failures in the log directly. Tracked as a follow-up, not blocking.
